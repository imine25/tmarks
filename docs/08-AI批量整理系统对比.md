# AI 批量整理系统对比

## 概述

TMarks 包含两套独立的 AI 批量整理系统，分别服务于不同的使用场景：

1. **Web 应用端（导入整理）** - 用于导入外部书签文件时的 AI 整理
2. **浏览器扩展端（工作区整理）** - 用于整理已有浏览器书签的 AI 整理

这两套系统虽然都使用 AI 技术，但目标、实现和使用场景完全不同。

## 系统对比表

| 特性 | Web 应用端（导入整理） | 浏览器扩展端（工作区整理） |
|------|----------------------|--------------------------|
| **使用场景** | 导入外部书签文件时 | 整理已有的浏览器书签 |
| **数据来源** | HTML/JSON 文件 | 浏览器书签 API |
| **AI 目标** | 为单个书签生成标签和描述 | 为整个书签集合生成文件夹结构 |
| **输入数据** | 书签列表（标题、URL、描述） | 域名统计（热度、路径、示例） |
| **输出结果** | 带 AI 标签的书签列表 | 域名到文件夹路径的映射 |
| **批量策略** | 每批 20 个书签 | 每批 300 个域名 |
| **处理粒度** | 书签级别 | 域名级别 |
| **文件夹操作** | 不涉及 | 创建/重组文件夹结构 |
| **标签操作** | 生成/优化标签 | 不涉及 |
| **描述操作** | 生成/优化描述 | 不涉及 |
| **热度分析** | 不涉及 | 支持历史记录热度分析 |
| **层级控制** | 不涉及 | 1-2 层文件夹限制 |
| **自定义规则** | 不支持 | 支持用户自定义分类规则 |
| **实时监控** | 进度条 | 实时日志控制台 |
| **数据持久化** | 导入到数据库 | 写入浏览器书签 |
| **代码位置** | `tmarks/src/components/import-export/` | `tab/src/background/handlers/ai-organize/` |

## 使用建议

### Web 应用端（导入整理）

**适合场景**
- 首次导入大量书签
- 从其他工具迁移到 TMarks
- 需要为每个书签生成精确标签
- 希望优化书签描述信息

**优势**
- 为每个书签生成精确标签
- 可选生成描述信息
- 支持预览和手动编辑
- 导入前完成整理


### 浏览器扩展端（工作区整理）

**适合场景**
- 定期整理已有书签
- 书签积累过多需要重组
- 希望按使用频率智能分类
- 需要合并相似域名的书签

**优势**
- 按使用频率智能分类
- 高频网站自动放到一级目录
- 支持历史记录热度分析
- 支持用户自定义规则

## 推荐工作流程

### 1. 初次使用

```
外部书签文件（HTML/JSON）
    ↓
Web 应用端 AI 整理
    ├─ 为每个书签生成标签
    ├─ 优化描述信息
    └─ 预览和编辑
    ↓
导入到数据库
    ↓
浏览器扩展同步
    ↓
保存到浏览器书签
```

### 2. 日常使用

```
在浏览器中保存书签
    ↓
使用 NewTab 快捷方式访问
    ↓
积累使用数据（点击次数、访问时间）
```

### 3. 定期整理

```
浏览器书签（已积累热度数据）
    ↓
浏览器扩展端 AI 整理
    ├─ 分析域名热度
    ├─ 生成文件夹结构
    └─ 重组书签位置
    ↓
优化后的文件夹结构
```

# Web 应用端（导入整理）详解

## 功能概述

在 Web 应用中导入书签文件时，可以使用 AI 批量整理功能，自动为每个书签生成智能标签和优化描述信息。

## 使用场景

- 从浏览器导出的 HTML 书签文件
- 从其他书签管理工具导出的 JSON 文件
- 批量导入大量书签时需要自动分类
- 希望为书签添加结构化标签


## 整理流程

```
1. 上传书签文件（HTML/JSON）
   ↓
2. 前端解析书签数据
   - HTML: 使用 DOMParser 解析
   - JSON: 直接 JSON.parse
   ↓
3. 用户选择是否启用 AI 整理
   ↓
4. [可选] AI 批量整理步骤
   ├─ 配置 AI 选项
   │  - 生成标签（必选）
   │  - 生成描述（可选）
   │  - 标签规范化（可选）
   ├─ 输入临时 API Key
   ├─ 估算 Token 消耗和费用
   └─ 开始 AI 整理
   ↓
5. AI 预览步骤
   ├─ 对比原始标签 vs AI 标签
   ├─ 查看 AI 生成的描述
   ├─ 手动编辑标签
   └─ 选择使用原始标签或 AI 标签
   ↓
6. 确认导入
   ↓
7. 批量保存到数据库
```

## 核心实现

### 文件解析（前端）

#### HTML 解析
- 使用 DOMParser 解析 HTML 结构
- 递归遍历 DL/DT 节点
- 提取书签属性：
  - HREF: URL
  - 文本内容: 标题
  - TAGS 属性: 标签
  - ADD_DATE: 创建时间
  - DD 节点: 描述
- 构建文件夹路径（从 H3 节点）

#### JSON 解析
- 支持 TMarks 格式（bookmarks 数组）
- 支持简单数组格式
- 字段映射：
  - title/name → title
  - url/href → url
  - tags → tags
  - description → description
  - folder → folder


### AI 整理逻辑

#### Prompt 构建

**系统提示词**
```
你是一个书签整理专家。请分析以下书签列表，为每个书签生成 2-5 个简洁的标签。

要求：
1. 标签要简洁，每个标签 2-5 个汉字或常见英文单词
2. 标签要通用、易于分类，避免过于具体
3. 优先使用已有标签列表中的标签
4. 如果书签有文件夹信息，可以参考但不要直接使用长路径作为标签

已有标签：[前 100 个现有标签]

书签列表：
1. 标题: xxx
   URL: xxx
   文件夹: xxx

请以 JSON 格式返回，格式如下：
{
  "results": [
    {
      "index": 1,
      "tags": ["标签1", "标签2"],
      "description": "简短描述"  // 可选
    }
  ]
}
```

#### 批量处理
- 每批 20 个书签（可配置）
- 分批调用 AI API
- 逐批收集结果
- 失败批次保留原始数据

#### 响应解析
- 提取 JSON 内容（支持 Markdown 代码块）
- 按 index 匹配书签
- 保存 AI 标签到 `ai_tags` 字段
- 保存 AI 描述到 `ai_description` 字段
- 保留原始标签到 `original_tags` 字段

### 预览与编辑

#### 对比显示
- 原始标签 vs AI 生成标签
- 原始描述 vs AI 生成描述
- 支持展开/折叠详情
- 显示 AI 标记（Sparkles 图标）

#### 编辑功能
- 手动编辑 AI 标签
- 恢复使用原始标签
- 实时保存修改
- 批量确认导入

#### 统计信息
- 总书签数
- AI 标签生成数
- 新增标签数
- Token 消耗统计


## 配置选项

### AI 整理选项
- **generateTags**：是否生成标签（必选）
- **generateDescription**：是否生成描述（可选）
- **normalizeTags**：是否规范化标签（可选）
- **existingTags**：现有标签列表（用于优先匹配）
- **batchSize**：批量大小（默认 20）

### 导入选项
- **skip_duplicates**：跳过重复书签
- **create_missing_tags**：自动创建缺失标签
- **preserve_timestamps**：保留时间戳
- **folder_as_tag**：将文件夹名作为标签（仅 HTML）

## Token 估算

### 估算公式
```typescript
基础 Token = 书签数量 × 80
// 每个书签约 50 tokens 输入 + 30 tokens 输出

如果生成描述：
总 Token = 基础 Token × 1.5
```

### 费用估算
各提供商价格（每 1M tokens）：
- OpenAI (gpt-4o-mini): $0.15
- Claude (claude-3-haiku): $0.25
- DeepSeek: $0.07
- 智谱 AI: $0.05
- ModelScope: $0.02
- SiliconFlow: $0.03
- iFlow: $0.10
- 自定义: $0.10

# 浏览器扩展端（工作区整理）详解

## 功能概述

AI 智能整理是浏览器扩展端的核心功能，可以自动分析所有书签，按照语义、使用频率和用户规则重新组织文件夹结构。

## 使用场景

- 浏览器书签杂乱无章
- 需要按主题重新分类
- 希望高频网站放在一级目录
- 需要合并相似域名的书签
- 定期整理书签结构


## 整理流程

```
1. 数据收集阶段
   ├─ 收集所有浏览器书签
   ├─ 按域名聚合统计
   ├─ 加载快捷方式使用数据
   └─ [可选] 读取历史记录
   ↓
2. 数据分析阶段
   ├─ 计算域名访问频率
   ├─ 计算最近访问时间
   ├─ 生成热度评分
   └─ 提取原始路径分布
   ↓
3. 生成域名摘要
   ├─ 域名基本信息
   ├─ 热度评分
   ├─ 原始路径
   └─ 示例书签
   ↓
4. AI 分析阶段
   ├─ 输入域名摘要
   ├─ 输入用户规则
   ├─ 输入现有文件夹
   └─ 输出整理方案
   ↓
5. 执行整理阶段
   ├─ 清空工作区
   ├─ 创建文件夹结构
   └─ 按方案复制书签
```

## 数据收集

### 域名摘要结构
```typescript
interface DomainSummary {
  domain: string              // 域名
  count: number               // 书签数量
  bookmarkCount: number       // 书签数量
  shortcutCount: number       // 快捷方式数量
  clickCount: number          // 点击次数
  historyCount: number        // 历史访问次数
  historyLastVisit: number    // 最后访问时间戳
  historyLastVisitISO: string // 最后访问时间 ISO
  recencyScore: number        // 最近访问评分 (0-1)
  heatScore: number           // 热度评分
  samples: Array<{            // 示例书签
    title: string
    url: string
    path: string
  }>
  originalPaths: Array<{      // 原始路径分布
    path: string
    count: number
  }>
}
```

### 收集流程

1. **收集浏览器书签**
   - 使用 `chrome.bookmarks.getTree()` API
   - 递归遍历书签树
   - 提取书签信息和路径

2. **按域名聚合**
   - 从 URL 提取域名
   - 统计每个域名的书签数量
   - 记录每个域名的原始路径分布

3. **加载使用统计**
   - 从 NewTab 快捷方式读取点击次数
   - 统计每个域名的快捷方式数量
   - 计算总点击次数

4. **读取历史记录**（可选）
   - 使用 `chrome.history.search()` API
   - 统计最近 N 天的访问记录
   - 按域名聚合访问次数和最后访问时间


## 热度计算

### 评分算法
```typescript
热度评分 = 书签数量 × 1.0
         + 点击次数 × 0.5
         + 历史访问次数 × 0.3
         + 最近访问评分 × 2.0

最近访问评分 = max(0, min(1, 1 - (当前时间 - 最后访问时间) / 分析天数))
```

### 评分范围
- 最低：0 分
- 最高：无上限（取决于使用频率）
- 用于排序和优先级判断

### 排序规则
1. 按热度评分降序
2. 热度相同时按书签数量降序

## AI Prompt

### 系统提示词结构

**角色定义**
- 专业的书签整理助手

**整理原则**

1. **聚合原则**
   - 同产品不同子域名合并（如 github.com, gist.github.com）
   - 功能相似的网站合并（如多个设计工具网站）
   - 少于 3 个域名的分类合并到更大类别

2. **层级规则**
   - 只允许 1-2 层文件夹
   - 禁止三层嵌套
   - 一级目录数量限制（可配置 2-7 个）

3. **优先级**（从高到低）
   - 用户自定义规则（最高优先级）
   - 历史访问热度（Top N 放一级目录）
   - 现有文件夹路径（优先保留）
   - AI 语义分析

4. **特殊要求**（根据配置）
   - 是否禁止新建一级目录（strictHierarchy）
   - 是否禁止新建文件夹（allowNewFolders）
   - 是否优先保留原路径（preferOriginalPaths）

### 输入信息
```typescript
{
  totalDomains: number,           // 总域名数
  totalBatches: number,           // 总批次数
  batchSize: number,              // 批量大小
  topHistoryLimit: number,        // Top N 高频域名
  topHistoryDomains: DomainSummary[], // Top N 域名列表
  folderPaths: Array<{            // 现有文件夹路径
    path: string,
    count: number
  }>,
  strictHierarchy: boolean,       // 是否禁止新建一级目录
  allowNewFolders: boolean,       // 是否允许新建文件夹
  preferOriginalPaths: boolean,   // 是否优先保留原路径
  batches: Array<{                // 域名批次
    batchIndex: number,
    totalBatches: number,
    waitForMoreBatches: boolean,
    isFinalBatch: boolean,
    domains: DomainSummary[]
  }>
}
```

### 输出格式
```json
{
  "domainMoves": [
    {
      "domain": "github.com",
      "path": "开发/代码托管"
    },
    {
      "domain": "stackoverflow.com",
      "path": "开发/问答社区"
    }
  ],
  "fallbackPath": "其他/未分类"
}
```


## 配置选项

### 历史热度分析
- **enableHistoryHeat**：是否启用历史热度分析
- **historyDays**：分析天数（7-90 天，默认 30）
- **historyHeatTopN**：Top N 高频域名放一级目录（10-50，默认 20）

### 层级控制
- **topLevelCount**：一级目录数量限制（2-7，默认 5）
- **strictHierarchy**：是否禁止新建一级目录（默认 false）
- **allowNewFolders**：是否允许新建文件夹（默认 true）
- **preferOriginalPaths**：是否优先保留原路径（默认 true）

### 自定义规则
- **rules**：用户自定义分类规则（文本）
  - 示例："GitHub 相关放到「开发/代码托管」"
  - 示例："设计工具放到「设计」"
  - 示例："新闻网站放到「资讯」"

### 自定义 Prompt
- **enableCustomPrompt**：是否使用自定义 Prompt
- **customPrompt**：自定义 Prompt 模板
- 支持变量替换：
  - `{{rules}}` - 用户规则
  - `{{domainSummariesJson}}` - 域名摘要 JSON
  - `{{topLevelCount}}` - 一级目录数量
  - `{{topHistoryLimit}}` - Top N 限制

### 其他选项
- **maxBookmarks**：域名批量大小（50-2000，默认 300）
- **verboseLogs**：是否显示详细日志（默认 true）

## 执行整理

### 整理流程

**1. 清空工作区**
- 获取 TMarks 根文件夹 ID
- 保留 "NewTab Home" 文件夹
- 删除其他所有子文件夹和书签
- 原始书签保留在浏览器其他位置

**2. 创建文件夹结构**
- 遍历 AI 规划的所有路径
- 按路径层级创建文件夹
- 使用 Map 缓存路径到 ID 的映射
- 避免重复创建

**3. 复制书签**
- 遍历所有原始书签
- 提取域名
- 查找 AI 规划的目标路径
- 在目标文件夹创建书签副本
- 每 100 个书签报告一次进度

### 一级目录限制

如果 AI 返回的一级目录数量超过限制：
1. 统计每个一级目录的域名数量
2. 按域名数量排序
3. 保留前 N 个最大的一级目录
4. 将超出的一级目录下的域名合并到 fallbackPath
5. 记录合并日志

### 错误处理
- TMarks 文件夹不存在时抛出错误
- 创建文件夹失败时记录日志
- 复制书签失败时跳过该书签
- 保证部分成功不影响整体

# 总结

## 两套系统的协同

这两套 AI 批量整理系统虽然独立，但可以很好地协同工作：

1. **Web 应用端**负责导入时的精细化标签生成

# 迁移计划：Web 端导入功能迁移到浏览器扩展端

## 迁移方案：完全迁移（方案 A）

### 迁移目标

将 Web 应用端的批量导入功能完全迁移到浏览器扩展端，统一书签管理入口，提供更好的用户体验。

### 迁移优势

1. **统一入口** - 用户只需在浏览器扩展中完成所有书签管理操作
2. **离线能力** - 可以在本地完成解析和 AI 整理，不依赖网络
3. **更好的性能** - 直接操作浏览器书签 API，无需通过服务器中转
4. **简化架构** - 减少 Web 端的复杂度，专注于展示和查询
5. **灵活的导入方式** - 支持仅本地导入或同步到服务器

## 功能设计

### 功能位置

NewTab 设置页面新增"导入"标签页（ImportTab）

```

   ↓

   │  - 生成标签（必选）

   │  - 标签规范化（可选）
   ├─ 使用已配置的 AI 设置
   ├─ 估算 Token 消耗和费用
   └─ 开始 AI 整理
   ↓
5. AI 预览步骤（如果启用 AI）
   ├─ 对比原始标签 vs AI 标签
   ├─ 查看 AI 生成的描述
   ├─ 手动编辑标签
   └─ 选择使用原始标签或 AI 标签
   ↓
6. 选择导入方式
   ├─ 方式 1：仅导入到浏览器书签（TMarks 文件夹）
   └─ 方式 2：导入到浏览器 + 同步到 TMarks 服务器
   ↓
7. 执行导入
   ├─ 创建文件夹结构
   ├─ 创建书签
   └─ [可选] 调用 API 同步到服务器
   ↓
8. 显示导入结果
   - 成功数量
   - 失败数量
   - 错误详情
```

### 两种导入路径

#### 路径 1：仅导入到浏览器（离线模式）

**优势**：
- 完全离线操作

```typescript
export interface OrganizeOptions {
  generateTags: boolean
  generateDescription: boolean
  normalizeTags: boolean
  existingTags?: string[]
  batchSize?: number
}

export async function* organizeBookmarks(
  bookmarks: ParsedBookmark[],
  aiConfig: AIConfig,
  options: OrganizeOptions,
  onProgress?: (progress: OrganizeProgress) => void
): AsyncGenerator<OrganizeProgress, OrganizeResult> {
  const batchSize = options.batchSize || 20
  const batches = []
  
  // 分批处理
  for (let i = 0; i < bookmarks.length; i += batchSize) {
    batches.push(bookmarks.slice(i, i + batchSize))
  }
  
  const allOrganized: OrganizedBookmark[] = []
  
  for (const batch of batches) {
    const prompt = buildOrganizePrompt(batch, options)
    const result = await callAI({ ...aiConfig, prompt })
    const organized = parseOrganizeResponse(result.content, batch)
    allOrganized.push(...organized)
    
    yield { current: allOrganized.length, total: bookmarks.length, status: 'processing' }
  }
  
  return { bookmarks: allOrganized, newTags: [], tokensUsed: 0 }
}
```

```typescript
// tab/src/lib/import/browser-import.ts

export async function importToBrowser(
  bookmarks: ParsedBookmark[],
  targetFolderId: string,
  onProgress?: (current: number, total: number) => void
): Promise<ImportResult> {
  const folderCache = new Map<string, string>()
  let successCount = 0
  let failedCount = 0
  const errors: ImportError[] = []
  
  for (let i = 0; i < bookmarks.length; i++) {
    const bookmark = bookmarks[i]
    
    try {
      // 确保文件夹存在
      const parentId = bookmark.folder
        ? await ensureFolderPath(targetFolderId, bookmark.folder, folderCache)
        : targetFolderId
      
      // 创建书签
      await chrome.bookmarks.create({
        parentId,
        title: bookmark.title,
        url: bookmark.url
      })
      
      successCount++
    } catch (error) {
      failedCount++
      errors.push({
        index: i,
        bookmark,
        error: error instanceof Error ? error.message : 'Unknown error'
      })
    }
    
    onProgress?.(i + 1, bookmarks.length)
  }
  
  return { success: successCount, failed: failedCount, errors }
}
```

```typescript
// tab/src/lib/import/server-sync.ts

export async function syncToServer(
  bookmarks: ParsedBookmark[],
  options: ImportOptions
): Promise<ImportResult> {
  const token = await getAccessToken()
  
  const response = await fetch('/api/v1/import', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${token}`
    },
    body: JSON.stringify({
      format: 'json',
      content: JSON.stringify({
        version: '1.0.0',
        exported_at: new Date().toISOString(),
        bookmarks,
        tags: []
      }),
      options
    })
  })
  
  if (!response.ok) {
    throw new Error('Server sync failed')
  }
  
  return await response.json()
}
```

## 需要考虑的问题

### 1. 大文件处理

**问题**：浏览器扩展有内存限制（通常 < 100MB）

**解决方案**：
- 分批解析（每批 100 个书签）
- 流式处理
- 进度保存（支持中断恢复）

### 2. AI 配置管理

**问题**：Tab 端和 Web 端的 AI 配置可能不同

**解决方案**：
- 使用 Tab 端已有的 AI 配置
- 支持临时 API Key（可选）
- 配置验证和测试

### 3. 进度显示

**问题**：导入过程可能较长，需要实时反馈

**解决方案**：
- 分阶段进度显示
  - 解析进度
  - AI 整理进度
  - 导入进度
- 支持取消操作
- 错误实时显示

### 4. 错误处理

**问题**：多种错误场景需要处理

**解决方案**：
- 文件格式错误：提示用户检查文件
- AI 调用失败：降级到不使用 AI
- 网络错误：重试机制
- 部分失败：显示详细错误列表

### 5. 向后兼容

**问题**：Web 端的导入功能是否保留

**解决方案**：
- 短期：保留 Web 端功能，添加提示引导用户使用扩展
- 长期：逐步废弃 Web 端导入，仅保留展示功能
