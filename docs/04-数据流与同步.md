## 书签保存流程

### 完整流程

```
1. 用户点击保存按钮
   ↓
2. Content Script 提取页面信息
   - 标题、URL、描述
   - 缩略图、Favicon
   - 页面内容（前 500 字）
   ↓
3. [可选] AI 标签推荐
   - 调用 AI 提供商 API
   - 基于页面内容分析
   - 返回推荐标签列表
   ↓
4. 用户选择/添加标签
   ↓
5. Background 处理保存请求
   ↓
6. 调用 TMarks API
   - POST /api/v1/bookmarks
   - 携带 JWT Token 或 API Key
   ↓
7. API 处理
   - 验证用户身份
   - 检查 URL 去重
   - 保存到 D1 数据库
   - 创建标签关联
   ↓
8. 本地缓存更新
   - 保存到 IndexedDB
   - 更新标签缓存
   ↓
9. [可选] 创建快照
   - Content Script 捕获页面
   - 提取图片（单独存储）
   - 上传到 R2
   ↓
10. 返回成功响应
```

### 离线处理

#### 网络错误检测
- 检测 fetch 错误
- 检测网络连接错误
- 检测连接拒绝错误

#### 离线队列机制
- 保存失败的书签到本地队列
- 记录创建时间和重试次数
- 存储在 IndexedDB 的 pendingBookmarks 表

#### 自动同步流程
1. 读取所有待同步书签
2. 逐个尝试上传到服务器
3. 成功后从队列中删除
4. 失败时判断错误类型：
   - 网络错误：保留在队列，等待下次同步
   - 其他错误：标记失败，增加重试次数

#### 触发时机
- 扩展启动时
- 网络恢复时
- 每日定时同步（23:00）
- 用户手动触发

## NewTab 同步机制

### 实时双向同步

#### 监听机制
- 监听浏览器书签 API 的 5 个事件：
  - onCreated - 新建书签时触发
  - onRemoved - 删除书签时触发
  - onChanged - 修改书签时触发
  - onMoved - 移动书签时触发
  - onChildrenReordered - 排序变化时触发

#### 同步锁机制
- 使用 isSyncing 标志防止循环更新
- 当正在同步时，忽略新的事件
- 同步完成后释放锁

#### 更新流程
1. 检测到书签变化事件
2. 检查同步锁状态
3. 加锁并开始同步
4. 更新 NewTab 网格项
5. 释放锁

### 文件夹发现

#### 自动发现机制
- 遍历浏览器书签树
- 查找名为 "TMarks" 的文件夹节点
- 递归搜索所有层级
- 返回找到的文件夹 ID

#### 查找规则
- 文件夹名称必须是 "TMarks"
- 必须是文件夹类型（无 URL）
- 优先返回第一个匹配项

#### 使用场景
- NewTab 启动时
- 同步操作前
- 导入书签时
- AI 整理前

### 增量更新

#### 更新策略
- 只更新变化的部分，不是全量替换
- 对比现有数据和新数据
- 识别三种变化类型：新增、删除、修改

#### 对比流程
1. 获取 TMarks 根文件夹的所有书签
2. 转换为网格项格式
3. 读取当前存储的网格项
4. 执行差异对比（diff）
5. 生成变化列表

#### 变化类型
- **add** - 新增的网格项
- **remove** - 删除的网格项
- **update** - 修改的网格项

#### 应用变化
- 遍历变化列表
- 根据类型执行对应操作
- 更新本地存储
- 触发 UI 刷新

## 标签页组同步

### 保存流程

```
1. 用户点击收纳标签页
   ↓
2. 收集当前窗口标签页信息
   - 标题、URL、Favicon
   - 位置、是否置顶
   ↓
3. 用户选择收纳模式
   - 新建分组
   - 添加到现有分组
   - 保存到文件夹
   ↓
4. Background 处理请求
   ↓
5. 调用 TMarks API
   - POST /api/v1/tab-groups
   - POST /api/v1/tab-groups/:id/items/batch
   ↓
6. API 处理
   - 创建/更新标签页组
   - 批量添加标签页项
   - 保存到 D1 数据库
   ↓
7. [离线模式] 本地保存
   - 保存到 IndexedDB
   - 加入待同步队列
   ↓
8. 关闭已收纳的标签页
```

### 离线队列

#### 离线保存机制
- 生成本地临时 ID
- 保存到 IndexedDB 的 pendingTabGroups 表
- 标记为未同步状态（synced: false）
- 记录创建时间

#### 自动同步流程
1. 查询所有未同步的标签页组
2. 逐个尝试上传到服务器
3. 成功后：
   - 标记为已同步
   - 保存服务器返回的 ID
4. 失败时：
   - 网络错误：保留在队列
   - 其他错误：记录错误日志

#### 触发时机
- 扩展启动时
- 网络恢复时
- 每日定时同步
- 用户手动触发

## 快照同步

### 快照创建流程

```
1. 用户触发快照创建
   ↓
2. Content Script 捕获页面
   - 使用 SingleFile V2
   - CSS 内联
   - 提取图片（不内联）
   ↓
3. 图片处理
   - 计算 SHA-256 哈希
   - Base64 编码
   - 类型检测
   ↓
4. 上传到服务器
   - POST /api/v1/bookmarks/:id/snapshots-v2
   - HTML 内容
   - 图片列表
   ↓
5. 服务端处理
   - 保存 HTML 到 R2
   - 图片去重检查
   - 保存新图片到 R2
   - 记录快照元数据到 D1
   ↓
6. 返回快照 ID
```

### 快照存储结构

```
R2 存储路径：
├── snapshots/
│   └── {userId}/
│       └── {bookmarkId}/
│           └── {version}.html
│
└── snapshot-images/
    └── {hash}.{ext}
```

### 去重机制

#### 图片去重流程
1. 计算图片的 SHA-256 哈希值
2. 在 snapshot_images 表中查询该哈希
3. 如果已存在：
   - 跳过上传
   - 直接关联到当前快照
4. 如果不存在：
   - 上传图片到 R2
   - 记录到数据库

#### 存储路径规则
- 图片路径：`snapshot-images/{hash}.{ext}`
- 使用哈希值作为文件名
- 自动识别图片类型（jpg, png, webp 等）

#### 优势
- 节省存储空间
- 减少上传时间
- 提高快照创建速度
- 全局去重（跨用户、跨书签）

## 缓存策略

### 扩展端缓存

#### IndexedDB 缓存结构
- **bookmarks 表**：书签数据缓存
  - 索引：userId, url, createdAt
- **tags 表**：标签数据缓存
  - 索引：userId, name
- **metadata 表**：元数据
  - 记录最后同步时间
  - 记录缓存版本

#### 缓存更新策略
1. 清空旧数据（避免冗余）
2. 批量插入新数据（提高性能）
3. 更新元数据（记录同步时间）
4. 使用事务保证原子性

#### 缓存失效
- 手动同步时清空重建
- 每日定时同步时更新
- 用户登出时清空

### 服务端缓存

#### KV 缓存机制（公开分享）
- 使用 Cloudflare KV 存储分享数据
- 缓存键格式：`share:{slug}`
- 缓存时长：1 小时（3600 秒）

#### 缓存流程
1. 检查 KV 中是否有缓存
2. 如果有缓存：直接返回
3. 如果无缓存：
   - 从 D1 数据库查询
   - 写入 KV 缓存
   - 返回数据

#### 缓存失效
- 分享内容更新时
- 分享被删除时
- 手动清除缓存时

#### 优势
- 减少数据库查询
- 提高响应速度
- 降低数据库负载
- 全球边缘节点加速

## 自动同步调度

### 定时任务

#### 每日同步（23:00）
- 计算到下一个 23:00 的时间差
- 使用 setTimeout 设置定时器
- 执行同步后重新调度下一次

#### 同步内容
1. 同步待处理书签
2. 同步待处理标签页组
3. 刷新本地缓存
4. 记录同步日志

#### 错误处理
- 捕获同步错误
- 记录错误日志
- 不影响下次调度

### 置顶书签刷新

#### 定时刷新机制
- 支持两个时间点：早上 8:00 或晚上 22:00
- 用户可在设置中选择刷新时间
- 计算到下一个刷新时间的延迟
- 使用 setTimeout 设置定时器

#### 刷新流程
1. 从 TMarks API 获取最新置顶书签
2. 更新本地 IndexedDB 缓存
3. 通知 NewTab Dock 栏刷新显示
4. 重新调度下一次刷新

#### 应用场景
- NewTab Dock 栏显示置顶书签
- 保持与服务器数据同步
- 减少实时请求频率

## 数据一致性

### 冲突解决

#### 时间戳优先策略
- 比较本地和远程数据的更新时间
- 选择时间戳较新的数据
- 适用于单向同步场景

#### 合并策略
- 保留远程数据为基础
- 合并本地的特定字段
- 适用于统计类数据：
  - 点击次数取最大值
  - 最后点击时间取最新值

#### 应用场景
- 离线编辑后同步
- 多设备同时修改
- 网络恢复后数据合并

### 事务处理

#### 原子操作保证
- 使用数据库事务确保数据一致性
- 多个操作要么全部成功，要么全部失败
- 避免部分数据写入导致的不一致

#### 典型场景
**保存书签并关联标签**
1. 开启事务
2. 保存书签到 bookmarks 表
3. 创建标签关联到 bookmark_tags 表
4. 提交事务

**批量更新标签**
1. 开启事务
2. 删除旧的标签关联
3. 创建新的标签关联
4. 提交事务

#### 错误回滚
- 任何步骤失败时自动回滚
- 保持数据库状态一致
- 返回错误信息给调用方

## 性能优化

### 批量操作

#### 批量保存
- 将大量数据分批处理
- 每批 100 条记录
- 使用 bulkAdd 提高性能
- 避免单条插入的开销

#### 批量更新标签
- 先删除旧的标签关联
- 再批量创建新的关联
- 使用事务保证原子性
- 支持多个书签同时更新

#### 性能优势
- 减少数据库操作次数
- 降低事务开销
- 提高写入速度
- 适合大量数据导入

### 增量加载

#### 分页加载机制
- 使用游标（cursor）实现分页
- 每页默认 50 条记录
- 按创建时间倒序排列

#### 加载流程
1. 构建查询（按 createdAt 倒序）
2. 如果有游标，过滤出更早的记录
3. 限制返回数量（pageSize）
4. 返回数据和下一页游标

#### 游标生成
- 使用最后一条记录的 createdAt 作为游标
- 如果记录数少于 pageSize，说明已到末尾
- 返回 null 表示没有更多数据

#### 应用场景
- 书签列表无限滚动
- 标签页组列表加载
- 历史记录查看

## 错误处理

### 重试机制

#### 指数退避策略
- 第 1 次重试：等待 1 秒
- 第 2 次重试：等待 2 秒
- 第 3 次重试：等待 4 秒
- 最多重试 3 次

#### 重试条件
- 只对网络错误进行重试
- 非网络错误直接抛出
- 避免无意义的重试

#### 应用场景
- API 请求失败
- 数据库连接失败
- 文件上传失败

#### 错误处理
- 记录每次重试的错误
- 达到最大重试次数后抛出最后的错误
- 返回给调用方处理

### 降级策略

#### AI 推荐降级
- 优先使用 AI 推荐标签
- AI 失败时使用关键词匹配
- 关键词匹配流程：
  1. 从页面内容提取关键词
  2. 获取用户现有标签列表
  3. 匹配包含关键词的标签
  4. 返回匹配结果

#### 降级触发条件
- AI API 请求失败
- AI 响应超时
- AI 返回格式错误
- API Key 无效

#### 用户体验
- 降级过程对用户透明
- 显示警告信息（可选）
- 保证功能可用性
- 记录降级日志
